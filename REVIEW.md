## 1. AI 모델 선정 및 시행착오 분석

### 1.1 요약 모델 (Summarization)
* **최종 선정**: `digit82/kobart-summarization`
* **후보 및 실패 사례**: `ainize/kobart-news`
* **교체 이유**:
    * **토크나이저 호환성**: 해당 모델은 구형 `transformers` 규격으로 제작되어 최신  환경에서 `AutoTokenizer` 로드 시 `RecursionError`(무한 재귀) 또는 클래스 매핑 오류가 발생함.
    * **유지보수 효율**: 에러 해결을 위해 최신 규격을 준수하여 `pipeline` API와 즉시 호환되는 검증된 모델로 교체함으로써 파이프라인의 **안정성**을 확보함.

### 1.2 감성 분석 모델 (Sentiment Analysis)
* **최종 선정**: `lxyuan/distilbert-base-multilingual-cased-sentiments-student`
* * **선정 이유 (Key Advantages)**:
    * **지식 증류 기반 효율성**: 대형 BERT 모델의 성능을 작은 모델로 전이시킨 'Student' 모델로, 성능 손실을 최소화하면서도 파라미터 수를  줄여 CPU 환경에서도  빠른 추론 속도를 보장함.
    * **다국어 대응력**: 한국어 전용 모델은 아니지만 다국어 데이터로 학습되어, 뉴스 내에 포함된 영문 고유명사나 외래어가 포함된 문장에서도 문맥 파악 능력이 우수함.
    * **정제된 레이블**: 긍정(Positive), 부정(Negative), 중립(Neutral)의 3단계를 명확히 구분하도록 훈련되어, 뉴스 기사의 어조(Tone)를 판별하는 ETL 파이프라인의 목적에 부합함.
* **후보 및 비교 분석**: 
    * `snunlp/KR-FinBert-SC`: 금융 뉴스 특화 모델로 설계되어 일반 뉴스 카테고리에 적용 시 범용성이 낮음을 확인.
    * `monologg/koelectra-base-v3-goemotion`: 정교한 감정 분류가 가능하나, 추론 속도 면에서 DistilBERT 기반의 경량 모델이 실시간 ETL 처리에 더 적합하다고 판단하여 최종 채택.

### 1.3 카테고리 분류 모델
* **최종 선정**: `jhgan/ko-sroberta-multitask`
* **선정 이유**: 뉴스 헤드라인과 본문의 의미적 유사도를 측정하여 정치/경제/사회 등의 카테고리로 매핑하는 데 있어, 한국어 문장 임베딩 성능이 검증된 SRoBERTa 구조를 채택하여 정확도를 높임.

* ## 2. Docker 인프라 및 네트워크 설계

### 2.1 서비스 포트 격리 및 내부 통신
* **설계 특징**: AI 서비스(FastAPI)의 포트를 호스트(Localhost)로 바인딩하지 않고 Docker 내부망에서만 노출함.
* **접근 가능 이유**: Docker Compose는 동일 네트워크 내의 컨테이너 간에 서비스 이름을 호스트명으로 사용하는 내장 DNS를 제공함. 따라서 `collector`는 `http://ai:8001`과 같은 주소로 `ai` 컨테이너에 즉시 접근 가능함.
* **의도**: 외부 사용자(User Client)가 AI 연산 서버에 직접 접근할 필요가 없으므로, 외부 공격 경로를 차단하기 위해 **포트 바인딩(`ports`) 대신 내부 노출(`expose`)만 설정**하여 보안성을 극대화함.

* ## 3. 데이터베이스 설계 및 예외 처리

### 3.1 UPSERT 도입
* **설계 의도**: RSS 피드 수집 특성상 동일한 기사가 중복으로 인입될 수 있음. 단순 `INSERT` 사용 시 기본키 충돌로 인해 전체 파이프라인이 중단되는 리스크 존재.
* **단순 INSERT와의 차이**: 
    - **INSERT**: 중복 발생 시 에러를 반환하며 트랜잭션 실패.
    - **UPSERT**: 중복 발생 시 기존 데이터를 유지(`DO NOTHING`)하거나 최신 분석 결과로 갱신(`DO UPDATE`)하여 에러 없이 처리 완료.
* **성과**: 수집 프로세스의 멱등성을 보장하여, 여러 번 실행해도 DB 상태가 안정적으로 유지되도록 설계함.
 
* ## 4. 인프라 자원 관리 및 가용성 설계

### 4.1 영속성 볼륨 활용
* **Model Cache 볼륨**: AI 분석에 사용되는 모델 파일을 호스트 디렉토리에 마운트하여 관리함. 볼륨이 없다면 컨테이너 재시작 시마다 모델을 재다운로드해야 하며, 이는 서비스 준비 시간 지연 및 불필요한 네트워크 트래픽 유발의 원인이 됨.
* **Postgres Data 볼륨**: 데이터베이스 컨테이너의 생명주기와 무관하게 수집된 뉴스 데이터 및 분석 결과를 보존함.

### 4.2 컨테이너 재시작 시나리오 분석
* **이슈**: 컨테이너는 기본적으로 휘발성 구조를 가짐. 
* **해결**: 볼륨 설계를 통해 컨테이너가 업데이트되거나 예기치 않게 종료된 후 재시작되어도, 기존 데이터와 학습된 자원을 즉시 재사용할 수 있는 상태 유지 환경을 구축함.
* **성과**: 장애 발생 시 시스템 복구 시간(RTO)을  단축하고 서비스 연속성을 확보함.

* ## 5. 예외 처리 전략: 결함 감내 설계

### 5.1 설계 결정: AI 분석 실패 시 NULL 허용 적재
* **전략 개요**: 특정 뉴스 기사의 요약, 감성 분석, 카테고리 분류 단계에서 AI 모델 서버 호출 실패나 추론 에러가 발생하더라도, 전체 ETL 파이프라인을 중단시키지 않고 분석 결과 컬럼에 `NULL`을 명시적으로 허용하여 DB에 저장함.

### 5.2 장단점 분석

####  장점: 시스템 가용성
* **전체 프로세스 보호**: RSS 피드에서 수집된 30건의 기사 중 단 1건에서만 AI 모델 오류가 발생해도 시스템 전체가 멈추는 리스크를 방지함. (1건의 실패가 29건의 성공을 방해하지 않음)
* **부분적 서비스 제공**: 분석이 안 된 기사라도 일단 원본 텍스트는 DB에 적재되므로, 유저는 최소한 기사 원문이라도 조회할 수 있는 상태를 유지함.

####  단점: 데이터 품질
* **후속 처리 필요**: DB 내에 분석 결과가 비어있는 데이터가 존재하게 되므로, API 서비스 레이어에서 사용자에게 보여줄 때 `NULL` 값을 필터링하거나 기본값을 처리하는 로직이 추가로 요구됨.
* **일관성 추적**: 어떤 기사가 분석에 실패했는지 별도의 로깅이나 모니터링 시스템을 통해 추적하고, 추후 재분석(Retry)을 수행하는 복구 프로세스가 필요함.
 
* ## 번외 도전과제 성능 최적화 및 병렬 처리 테스트

### 1 비동기 병렬 처리(Async/Await) 도입
* **최적화 전략**: `asyncio.gather`를 활용하여 10건의 뉴스 기사를 AI 서버에 동시 요청함으로써, 네트워크 대역폭 활용도를 높이고 전체 분석 시간을 단축하고자 함.
* **1차 테스트 결과**: 순차 처리 대비 약 6.1%의 미미한 성능 향상 확인. 원인 분석 결과 AI 서버의 단일 워커 병목으로 판명됨.

### 2 서버 Worker 증설 및 자원 경합 분석
* **개선 시도**: 서버의 처리 용량을 늘리기 위해 AI 서버의 워커 수를 1개에서 3개로 증설하여 물리적 동시성을 확보함.
* **최종 테스트 결과**: 
    - **개선 효율**: 8.9%로 소폭 상승.
    - **특이사항**: 전체 소요 시간은 워커 1개일 때보다 **약 2배 이상 증가**(21초 → 45초)하는 역설적인 결과 발생.
* **기술적 인사이트**: 
    - **자원 경합 발생**: 무거운 AI 모델들이 워커 수만큼 중복 로드되면서 CPU와 메모리 자원을 차지하기 위해 싸우는 'Resource Contention' 발생.
    - **결론**: CPU 집약적인 AI 연산 환경에서는 단순한 비동기/워커 증설보다 배치 처리나 GPU 가속과 같은 방법 고려.
